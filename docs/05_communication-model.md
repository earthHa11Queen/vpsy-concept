# 05. 通信・伝達モデル（TCP/IPアナロジー）

## 概要

VPSYでは、人間同士のコミュニケーションや情報伝達を  
**TCP/IPプロトコルの概念** を応用してモデル化します。

この応用により以下が実現されます。

- 複雑な情報を構造化して各処理層へ渡せる
- 既存のネットワーク技術・ライブラリをそのまま転用できる
- Dockerネットワークとの親和性が高く、集団拡張が自然に実現できる
- Zabbixなどの監視ツールとの連携が容易になる

---

## インプットデータの構造化（TCP/IPダイアグラム応用）

各種インプット（会話・行動・記憶・環境）を1つのデータ単位として扱います。

```
┌──────────────────────────────────────────┐
│ イーサネットヘッダ相当                       │
│  → 「会話なのか？行動なのか？」の簡略分類      │
├──────────────────────────────────────────┤
│ IPヘッダ相当（記憶ヘッダ）                   │
│  → 記憶テーブルからの参照値・バイアス情報       │
├──────────────────────────────────────────┤
│ TCPヘッダ相当（環境ヘッダ）                  │
│  → 環境テーブルからの固定バイアス情報          │
├──────────────────────────────────────────┤
│ アプリケーション層データ（ペイロード）          │
│  → 会話内容・行動内容                        │
│  　+ 状況データ（ヘッダとして付加）            │
└──────────────────────────────────────────┘
```

### 各処理層でのヘッダ処理フロー

```
情報入力（フルパケット）
  ↓
L2 観念層：イーサネットヘッダを処理 → 観念処理済ヘッダを付加
  ↓
L3 理解層：記憶ヘッダ＋観念処理済ヘッダを処理 → 理解処理済ヘッダを付加
  ↓
L4 傾向性層：環境ヘッダをメインに処理 → 感情マッピングへのルーティング
  ↓
L5 感情層：残りのデータをブルチック等の感情定義テーブルにマッピング
```

各層でヘッダを剥がしながら処理する構造は、  
TCP/IPのカプセル化・デカプセル化と同じ論理構造です。

## TTL（Time To Live）による感情の持続・消滅管理

感情データにTTLを付与し、IPパケットに類似した扱いをします。

```
感情データ（TTL付きIPパケット相当）
  ↓ 単位時間内にTTL枯渇 → 感情消滅
  ↓ TTLが気分単位時間まで継続 → 気分層へフェーズ移行
  ↓ 気分TTLがさらに継続 → 気持ち層へフェーズ移行
```

**サブネット固定化のアナロジー**

気分が一定期間持続すると、その単位時間内の「感情傾向性のサブネット部」が固定されます。  
これはIPv4のサブネットマスクで特定の範囲が固定されることと同じ論理です。

```
感情揺れ → 気分サブネットが確定 → 大きな情報量が来ない限りぶれない
```

## トーン（声・会話のトーン）のプロトコル化

会話のトーンを独自プロトコルとして定義します。

声のトーンは「明度・彩度の組み合わせ」に似た多次元パラメータとして扱います。

```
トーンデータセット
  ├── 声量（絶対値・相対値）
  ├── 語気の強弱
  ├── アクセントパターン
  ├── 文脈内の正常域からの偏差
  └── 受け手の閾値パラメータ
```

**トーンの意味区分**

| トーンの偏差 | 解釈 |
|------------|------|
| 正常域内 | 意味通信（内容で通信） |
| 正常域からやや離れたグレーゾーン | 感情通信（感情で通信） |
| 正常域から大きく外れる | 非理解状態・特性・状況データとして分類 |

## IPV4アドレス応用による分類管理

サブネットグループの概念を「データ分類」と見立てることで、  
環境データ・関係性データ・状態データなどをIPv4と同じ演算方式で管理できます。

```
会話サブネットグループ（例：192.168.1.0/24）
  ├── 会話内容データ
  ├── 状況データ（ヘッダ付加）
  └── トーンデータ

環境サブネットグループ（例：10.0.0.0/8）
  ├── 環境マスタ参照値
  └── 経時的な環境変化ログ
```

ルーティングテーブルの概念を用いることで、  
どの情報がどの処理モジュールへ渡されるかの制御が構造化されます。

## 集団拡張：Dockerネットワークによる個人間通信

個人をDockerコンテナとして実装し、コンテナ間ネットワークで接続することで  
集団の感情相互作用をシミュレートします。

```
[個人A コンテナ]──[Dockerネットワーク]──[個人B コンテナ]
       ↕                                        ↕
[個人C コンテナ]────────────────────────[個人D コンテナ]
```

**ネットワーク遅延・ジッターのパラメータ化**

国家間・文化間の感情伝達をシミュレートする際に、  
物理ネットワーク遅延を意図的に設定することで以下を表現できます。

| パラメータ | 表現する現象 |
|-----------|------------|
| ネットワーク遅延 | 情報伝達の物理的・心理的遅れ |
| ジッター | 翻訳時の誤訳・意味の変容 |
| パケットロス | コミュニケーションの断絶・誤解 |

## 生成AIとの連携による情報生成

シミュレーション用の情報データ（会話内容・行動内容）は、  
生成AIへ定型プロンプトを送信することで動的に生成します。

これにより、人の手によるデータ作成をほぼ不要にし、  
分類された状態での大量データ生成が可能となります。
