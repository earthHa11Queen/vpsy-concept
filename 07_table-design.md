# 07. DBテーブル設計案

## 概要

VPSYの感情シミュレーションを支えるデータベース設計の案です。  
現時点では概念設計レベルであり、詳細なカラム定義は実装フェーズで確定します。

---

## 各テーブルの設計考察

### 1. 短期記憶テーブル

気分フェーズに達した感情データが蓄積されるテーブルです。

- ハフマン符号化によって符号化されたビット列を格納
- 環境テーブルの参照によってバイアスが制限される
- 気分テーブルのリレーション先として機能

**主要カラム（案）**

| カラム名 | 型 | 説明 |
|---------|---|------|
| memory_id | INT | PK |
| person_id | INT | 対象個人ID |
| encoded_data | BYTEA | ハフマン符号化済みデータ |
| emotion_id | INT | 感情定義テーブルへのFK |
| occurred_at | TIMESTAMP | 発生時刻 |
| ttl_remaining | INT | 残TTL値 |
| compression_count | INT | 圧縮回数（忘却進行度） |

---

### 2〜4. 長期記憶・過去・昔の記憶テーブル

短期記憶テーブルと同様の基本構造を持ちますが、  
圧縮回数（`compression_count`）が増加するほど元データからの乖離が大きくなります。

**昔の記憶テーブルの特徴**

ライフステージ（幼少期・学童期・思春期・成人期など）や  
ライフイベント（進学・就職・死別など）単位でまとめて圧縮されます。

```
認知症・記憶喪失の表現：
  「昔の記憶テーブルしか参照できない状態」
  「昔の記憶テーブルが欠損している状態」
として実装可能
```

---

### 5〜6. 環境マスタ・環境テーブル

**環境マスタテーブル**

`06_data-types.md` で定義した環境データタイプのマスタです。

**環境テーブル**

個人の環境状態を時系列で記録します。  
カレント（現在）の環境と、それ以前の環境履歴の両方を管理します。

- カレント環境：現在の処理に直接作用
- 過去の環境：長期記憶・過去テーブルと合わさって圧縮される
- 環境の「克服」は、環境テーブルのフラグ更新として表現可能

---

### 7. 観念モジュールテーブル

観念処理ロジック（ブラックボックス関数）をDBから呼び出す仕組みです。

- モジュールの定義と処理結果を同レコードまたは関連レコードに保存
- これによりブラックボックス関数の「答え合わせ」が可能になる
- モジュールのバージョン管理を行うことで、観念の「成長」を表現できる

---

### 8. 理解力パラメータテーブル

理解力の個人差を数値パラメータとして管理します。

**変化履歴の記録**

パラメータ変化の経歴を記録することで、以下の判断基準を得られます。

- どのような成長経路が「リアリティのある成長」か
- パラメータ変化がシミュレーション結果に与える影響の検証

---

### 10. 感情定義テーブル

傾向性層から渡された値（整数値の範囲）を、感情名にマッピングするテーブルです。

- 範囲で生起感情が決まるため、1レコードではなく複数レコードが取得されうる設計
- ブルチックの感情の輪を初期データとして使用
- 独自の感情定義を追加可能な拡張性を持たせる

---

### 12〜13. 気分・気持ちテーブル

感情定義テーブルに近い構成ですが、以下の点で異なります。

| 項目 | 気分テーブル | 気持ちテーブル |
|-----|------------|--------------|
| 持続時間 | 時間〜日単位 | 週〜月以上 |
| リレーション | 短期記憶テーブルに作用 | 長期記憶テーブルに関与 |
| バイアス | 短期記憶・環境テーブルで制限 | 長期記憶に関与・定着 |

---

## 設計上の注意事項

- 全テーブルにわたって `person_id` によって個人単位のデータを完全に分離する
- 一時的なパラメータ計算には PostgreSQL のテンポラリテーブルを活用する
- 各テーブルのタイムスタンプカラムは `時刻T` 管理の基盤となるため必須
- 傾向性テーブル（#9）は現時点では詳細未定義（今後の考察対象）
